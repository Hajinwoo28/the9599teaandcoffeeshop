Security Features

01
CSRF Protection Flask-WTF · Cross-Site Request Forgery Defense
Free Critical
CSRF attacks trick authenticated users into unknowingly submitting malicious requests. Flask-WTF's CSRFProtect automatically generates and validates a signed token on every state-changing form submission (POST, PUT, DELETE). Without this, an attacker on another website could submit a form to your POS as the logged-in cashier.

Implementation Steps

  1. Install: pip install Flask-WTF
  2. Initialize: CSRFProtect(app) after app creation
  3. Set WTF_CSRF_ENABLED = True and a strong SECRET_KEY
  4. Add {{ csrf_token() }} hidden input to every HTML form
  5. For AJAX: include X-CSRFToken header in fetch/axios calls
  6. Set WTF_CSRF_TIME_LIMIT = 3600 (tokens expire in 1 hour)

Risks & Mitigations

⚠️
Risk: Breaks API endpointsJSON APIs don't send form tokens by default.
✅
Mitigation: Exempt API routesUse @csrf.exempt on stateless JSON routes protected by session auth.
⚠️
Risk: Token expiry logs users out mid-sessionLong-open tabs may exceed the time limit.
✅
Mitigation: Refresh token via JSPing /csrf-token on activity to refresh before expiry.
# In app.py from flask_wtf.csrf import CSRFProtect csrf = CSRFProtect(app) # In every HTML form template: <input type="hidden" name="csrf_token" value="{{ csrf_token() }}"> # For AJAX (JavaScript): fetch('/api/transaction', { method: 'POST', headers: { 'X-CSRFToken': getCsrfToken() }, body: JSON.stringify(cart) });
02
Rate Limiting Flask-Limiter · Brute-Force & DoS Prevention
Free Critical
Without rate limiting, an attacker can attempt thousands of passwords per second (brute-force), flood endpoints with requests (DoS), or enumerate valid usernames. Flask-Limiter enforces per-IP request ceilings on sensitive routes such as /login, /api/transaction, and /verify-2fa.

Implementation Steps

  1. Install: pip install Flask-Limiter
  2. Initialize with Limiter(app, key_func=get_remote_address)
  3. Apply global defaults: "300 per day", "60 per hour"
  4. Apply strict limits to login: @limiter.limit("10 per minute")
  5. Apply to 2FA: @limiter.limit("5 per minute")
  6. Return HTTP 429 with a clear message on limit exceeded

Risks & Mitigations

⚠️
Risk: Legitimate users get blockedStaff on the same IP as previous attacker may be locked out.
✅
Mitigation: Generous global limitsOnly login/sensitive routes use strict limits. Normal use is unaffected.
⚠️
Risk: NAT hides multiple usersA shared office IP may hit limits faster.
✅
Mitigation: Per-user limits in prodSwap get_remote_address for a logged-in user ID key function.
03
HTTP Security Headers Flask-Talisman · HSTS, CSP, X-Frame-Options & More
Free High
Flask-Talisman adds a suite of HTTP security response headers that instruct browsers to enforce security policies. This single library addresses six separate attack categories simultaneously — all for free — including protocol downgrade attacks, clickjacking, MIME-type confusion, and XSS via injected scripts.

Headers Added & What They Do

  • Strict-Transport-Security: Forces HTTPS for 1 year (HSTS)
  • Content-Security-Policy: Blocks scripts from unknown origins
  • X-Frame-Options: DENY: Prevents clickjacking in iframes
  • X-Content-Type-Options: Stops MIME-type sniffing
  • Referrer-Policy: Limits URL leakage to third parties
  • Permissions-Policy: Disables camera, mic, geolocation

Risks & Mitigations

⚠️
Risk: CSP breaks inline scriptsAny <script> tag in templates will be blocked.
✅
Mitigation: Move JS to .js filesAll JavaScript must be in external files referenced by script src.
⚠️
Risk: HSTS cannot be undoneIf SSL cert expires, users cannot access the site at all.
✅
Mitigation: Use auto-renew certsDeploy with Let's Encrypt (Certbot) for automatic SSL renewal.
04
Audit Trail Logging Python logging + SQLAlchemy · Full Forensic Record
Free High
Every security-relevant action — logins, failed attempts, product changes, transactions, access denials — is logged with a timestamp, username, IP address, and event type. Logs are stored both in rotating files (for off-system backup) and a dedicated database table (for in-app audit review). This is essential for incident response and regulatory compliance.

Implementation Steps

  1. Create an AuditLog database model with timestamp, event_type, user_id, ip_address, details, severity
  2. Create a RotatingFileHandler (10 MB max, 5 backups) targeting logs/audit.log
  3. Write a central audit(event, details, severity) helper function
  4. Call audit() on: login success/fail, logout, product CRUD, transaction creation, access denied, account lockout
  5. Create /audit-logs route (admin-only) for viewing the log database
  6. Set log directory permissions to restrict access (Linux: chmod 640)

Events Logged

  • LOGIN_OK / LOGIN_FAIL / LOGIN_BLOCKED
  • LOGOUT / 2FA_OK / 2FA_FAIL
  • ACCOUNT_LOCKED / ACCOUNT_UNLOCKED
  • PRODUCT_ADD / EDIT / DELETE
  • TXN_CREATED
  • USER_ADD / USER_DEACTIVATED
  • AUTHZ_DENIED / RATE_LIMITED
  • PW_CHANGED / 2FA_ENROLLED
05
Input Validation & XSS Prevention bleach · Sanitize All User-Supplied Data
Free Critical
Cross-Site Scripting (XSS) allows attackers to inject malicious JavaScript into pages viewed by other users. A cashier entering <script>...</script> in a product name field could compromise every manager's session. The bleach library strips all HTML tags from user inputs before they are stored or displayed. Combined with SQLAlchemy's parameterized queries, this also prevents SQL injection.

Implementation Steps

  1. Install: pip install bleach
  2. Create a central _sanitize(text, max_len) helper
  3. Run every request.form.get() and request.args.get() through _sanitize() before use
  4. Never pass raw user input to database queries — always use SQLAlchemy ORM
  5. Validate numeric fields with explicit float() / int() casting
  6. Whitelist allowed values for role, payment_method using Python sets

What the Sanitizer Does

import bleach def _sanitize(text, max_len=255): # Strip ALL HTML tags clean = bleach.clean( str(text), tags=[], # no tags allowed strip=True # remove, don't escape ) return clean[:max_len] # Input: "<script>evil()</script>" # Output: "" (tag stripped) # Input: "Coca Cola 1L & 2L" # Output: "Coca Cola 1L &amp; 2L" (safe)
06
Account Lockout Policy Custom Implementation · Max Attempts + Timed Lockout
Free High
Even with rate limiting, a slow persistent attacker could eventually guess a weak password. Account lockout is a secondary defense: after 5 consecutive failed login attempts, the account is locked for 15 minutes automatically. The lockout state is stored in the database and is not dependent on the browser session.

Implementation Steps

  1. Add failed_attempts, is_locked, locked_until columns to the User model
  2. On each failed login, call user.record_failed_attempt()
  3. Lock account when failed_attempts >= MAX_LOGIN_ATTEMPTS (default: 5)
  4. Set locked_until = now + 15 minutes on lock
  5. On each login attempt, call user.is_account_locked() first
  6. Auto-unlock by checking if locked_until has passed
  7. Add admin route /users/<id>/unlock for manual override

Risks & Mitigations

⚠️
Risk: Denial of service via lockoutsAttacker deliberately locks legitimate accounts.
✅
Mitigation: Auto-unlock + admin override15-minute auto-unlock. Admin can immediately unlock any account.
⚠️
Risk: Exposing lockout stateError message reveals if a username is valid.
✅
Mitigation: Generic error messagesShow "Invalid credentials" even for locked accounts to prevent enumeration.
07
Two-Factor Authentication (2FA) pyotp · TOTP Compatible with Google/Microsoft Authenticator
Free High
Even if a password is compromised through phishing or data breach, 2FA ensures the attacker still cannot log in without also having the user's physical phone. The pyotp library generates TOTP (Time-based One-Time Password) secrets compatible with Google Authenticator, Microsoft Authenticator, and Authy — all free apps available on any smartphone.

Implementation Steps

  1. Install: pip install pyotp qrcode[pil] Pillow
  2. Add totp_secret and totp_enabled columns to User model
  3. Create /setup-2fa route that generates a QR code
  4. On login success: if user.totp_enabled, redirect to /verify-2fa
  5. Store pre-2FA user ID in session (session['_2fa_uid'])
  6. Verify token with pyotp.TOTP(secret).verify(token, valid_window=1)
  7. Only call login_user() after successful 2FA verification

Setup Flow for Users

  1. User goes to Profile → Enable 2FA
  2. System generates a secret key and displays a QR code
  3. User scans QR code with their Authenticator app
  4. User enters the 6-digit code shown in the app to confirm
  5. 2FA is now active — required on every login
  6. On each login: enter password → enter 6-digit code from app

Compatible apps: Google Authenticator, Microsoft Authenticator, Authy (all free)

08
Secure Session Management Flask Configuration · HTTPOnly, SameSite, Expiry & Strong Protection
Free Medium
Session cookies are the keys to the kingdom — if stolen, an attacker can impersonate any user without knowing their password. Flask's built-in session system becomes secure through proper configuration: marking cookies as HTTPS-only, inaccessible to JavaScript, and short-lived. Flask-Login's session_protection = 'strong' mode also invalidates sessions if the user's IP or browser fingerprint changes mid-session.

Configuration Settings

app.config.update( # Only sent over HTTPS (no plain HTTP) SESSION_COOKIE_SECURE=True, # JS cannot read the cookie (XSS protection) SESSION_COOKIE_HTTPONLY=True, # Blocks cross-origin cookie submission SESSION_COOKIE_SAMESITE='Lax', # Sessions expire after 8 hours PERMANENT_SESSION_LIFETIME= timedelta(hours=8), ) # Invalidate on IP/UA change: login_manager.session_protection = 'strong'

What Each Setting Prevents

🔒
SESSION_COOKIE_SECUREPrevents interception over unencrypted Wi-Fi (man-in-the-middle)
🔒
SESSION_COOKIE_HTTPONLYPrevents XSS attacks from reading the session token
🔒
SESSION_COOKIE_SAMESITESecondary CSRF defense — blocks cross-site cookie use
🔒
session_protection='strong'Detects session hijacking by IP/browser fingerprint change

  • ✓ Install SSL certificate (free via Let's Encrypt / Certbot)
  • ✓ Set force_https=True in Flask-Talisman configuration
  • ✓ Restrict logs/ directory permissions: chmod 640 logs/audit.log
